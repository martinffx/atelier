# Code Review: React useEffect data fetching

## Summary
This React useEffect implements a basic data fetching pattern with async/await. While functional, the implementation lacks critical error handling and edge case management that could cause runtime issues in production.

## Findings

### Critical (Fix before merge)
- **Issue:** No error handling for failed network requests
- **Location:** useEffect (lines 1-8)
- **Impact:** Uncaught promise rejection when fetch fails. If `/api/data` returns 404, 500, or is unreachable, the app will crash with an unhandled promise rejection. Users see no feedback when things go wrong.
- **Suggestion:** Wrap in try/catch and handle errors:
```javascript
try {
  const response = await fetch('/api/data');
  if (!response.ok) throw new Error(`HTTP ${response.status}`);
  const json = await response.json();
  setData(json);
} catch (error) {
  setError(error.message);
}
```

### Critical (Fix before merge)
- **Issue:** No response validation before JSON parsing
- **Location:** Line 4
- **Impact:** If server returns non-JSON (e.g., 500 HTML error page), `response.json()` throws. Could happen if API returns error HTML instead of JSON.
- **Suggestion:** Check `response.ok` before parsing:
```javascript
if (!response.ok) throw new Error(`API error: ${response.status}`);
```

### High Priority (Should address)
- **Issue:** No cleanup for component unmount
- **Location:** useEffect (lines 1-8)
- **Impact:** If component unmounts before fetch completes, calling `setData` on unmounted component causes "Can't perform a React state update on an unmounted component" warning. Memory leak potential.
- **Suggestion:** Use AbortController:
```javascript
useEffect(() => {
  const controller = new AbortController();
  const fetchData = async () => {
    try {
      const response = await fetch('/api/data', { signal: controller.signal });
      const json = await response.json();
      setData(json);
    } catch (e) {
      if (e.name !== 'AbortError') setError(e.message);
    }
  };
  fetchData();
  return () => controller.abort();
}, []);
```

### Medium Priority (Nice to have)
- **Issue:** No loading state indication
- **Location:** useEffect (lines 1-8)
- **Impact:** User sees nothing while waiting for API response. Poor UX — users may think app is broken or click repeatedly.
- **Suggestion:** Add loading state alongside data:
```javascript
const [loading, setLoading] = useState(true);
// Set loading false in catch block too
```

### Low Priority (Future consideration)
- **Issue:** Hardcoded API endpoint
- **Location:** Line 2
- **Impact:** Not configurable per environment. Would require code changes to switch between dev/staging/prod APIs.
- **Suggestion:** Move to environment variable: `process.env.REACT_APP_API_URL`

### Positive Findings
- Correct use of async/await syntax
- Empty dependency array `[]` is appropriate for one-time fetch
- Clean function naming (`fetchData` describes intent)
- Modern React pattern with useEffect for side effects

## Recommendations
1. Add try/catch error handling immediately (Critical)
2. Validate `response.ok` before JSON parsing (Critical)
3. Implement AbortController for unmount safety (High Priority)
4. Consider adding loading state for better UX (Medium Priority)
5. Extract API URL to environment config (Low Priority)

The most important fixes are #1 and #2 — without error handling, this code will crash in production when network requests fail.
